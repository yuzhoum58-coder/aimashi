## 一、总体架构

```
┌──────────────────────────────────────────────────┐
│              PySide6 主窗口 (MainWindow)           │
│  ┌──────────────────────┬──────────────────────┐ │
│  │   四视图 MPR+3D      │   右侧控制面板        │ │
│  │  ┌──────┐ ┌──────┐  │  ┌──────────────┐   │ │
│  │  │Axial │ │Sagit │  │  │ 手动配准 6DOF │   │ │
│  │  ├──────┤ ├──────┤  │  │ Tx Ty Tz     │   │ │
│  │  │Coron │ │  3D  │  │  │ Rx Ry Rz     │   │ │
│  │  └──────┘ └──────┘  │  │ 矩阵显示      │   │ │
│  │  跨视口联动 (十字线)   │  └──────────────┘   │ │
│  └──────────────────────┴──────────────────────┘ │
├──────────────────────────────────────────────────┤
│            Python 数据处理层                        │
│  pydicom → SimpleITK → numpy → VTK               │
│  (DICOM解析) (体数据处理) (配准矩阵) (GPU渲染)    │
└──────────────────────────────────────────────────┘
```

## 三、关键技术决策（需要讨论）

### 决策1：四视图的交互模式

有两个方案，需要你定：

**方案A：一个Interactor + 手动事件路由**
- 优点：架构简洁，一个鼠标事件循环
- 缺点：每个视口独立旋转/缩放需要自己实现事件分发
- 复杂度：⭐⭐⭐

**方案B：四个Interactor（每个视口一个）**
- 优点：每个视口独立VTK交互器，原生支持旋转/缩放/平移
- 缺点：VTK的QVTKRenderWindowInteractor不支持多套，需要hack（创建4个RenderWindow或者4个vtkViewport）
- 复杂度：⭐⭐⭐⭐⭐

**建议：** 方案A。先做一个Interactor，自己处理视口的旋转/缩放事件分发。3D视图可以独立一个Interaction模式。

### 决策2：叠加渲染的方式

**方案A：双Volume叠加（GPU体绘制）**
- CT和MRI各自一个vtkVolume，VTK混合渲染
- 优点：专业效果好，透明度连续可调
- 缺点：GPU体绘制消耗大（两倍显存）

**方案B：2D切片的颜色融合**
- MPR切片层面做颜色叠加：CT→灰色，MRI→红色通道
- 优点：轻量快速，4G显存也能跑
- 适合：手动配准视觉反馈

**建议：** 方案B做配准时的实时反馈（快），方案A做最终展示效果——两种都支持，用切换开关选择。

### 决策3：坐标空间对齐

一个容易忽略的坑：CT和MRI的DICOM坐标可能不一致。
- ImagePositionPatient可能不同（扫描位置不同）
- PixelSpacing可能不同（CT: 0.5-1.0mm, MRI: 0.5-1.5mm）
- 方向矩阵可能不同

**方案：** 加载时统一用SimpleITK重采样到同一空间（以CT为参考），确保初始对齐。

## 五、风险和应对

| 风险 | 概率 | 影响 | 应对 |
|:----|:---:|:----:|:----|
| VTK+PySide6版本兼容性问题 | 中 | 高 | pip安装官方二进制包，不源码编译 |
| 多volume叠加渲染性能（显存不足） | 中 | 中 | 先做2D切片融合方案B，降显存需求 |
| DICOM序列过大（>2GB） | 低 | 中 | 分块加载+降采样 |
| 打包体积过大（>300MB） | 高 | 低 | UPX压缩，可接受 |
